# User Registration

## What it does
Handles new user sign-up (full registration), SSO-based partial registration (completing profile
after social login), OAuth2 login/logout/token-refresh flows, and B2B invitation code validation.
UserApi intermediates between the frontend, IDP (OpenID Connect), Hybris, and reCAPTCHA.

## Architecture

| Layer | Component | Role |
|---|---|---|
| .NET service | `GroheNeo.UserApi` | Registration, attribute update, OAuth2 token exchange |
| .NET service | `GroheNeo.RecaptchaApi` | reCAPTCHA v3 validation (shared with Forms) |
| Next.js routes | `/api/auth/`, `/api/user/validate-invitation-code` | OAuth2 callback, login initiation |
| External | IDP (OpenID Connect) | User directory, token issuance |
| External | Hybris (SAP Commerce) | User account (mirrored from IDP) |
| External | Google reCAPTCHA v3 | Form spam protection |

## Data Flow

### Full registration
```
Browser (registration form)
    → POST /v1/register  { userAttributes, neoAttributes, captchaResponse }
        → RecaptchaApi: validate token
        → IDP: create user
        → return 201 Created
```

### SSO login + partial registration
```
Browser → GET /api/auth/login?locale=...
    → Next.js generates state (CSRF), redirects to IDP auth URL

IDP → GET /api/auth/callback?code=...&state=...
    → Next.js validates state
    → POST /v1/oauth2/login { code }
        → IDP: exchange code for tokens
        → returns 200 (complete) or 206 (partial registration needed)
    → if 206: redirect to partial-registration page
    → if 200: store tokens in cookies, redirect to referrer
             + merge anonymous cart into user cart
```

### Partial registration
```
Browser (partial-reg page) → POST /v1/register/partial { captchaResponse, neoAttributes }
    → JWT required (user already has tokens from SSO)
    → RecaptchaApi: validate
    → IDP: PATCH user attributes
    → return 201
```

## Key Business Logic

- **reCAPTCHA required** for: full registration, partial registration, service request form,
  contact form. NOT required for: request quote, support/aftersales.
- **reCAPTCHA bypass:** `X-Bypass-Captcha` header + matching env var `X-Bypass-Captcha-Key` —
  dev/test only. Returns 403 Forbidden if captcha fails and no bypass.
- **Partial registration flag (206):** IDP returns 206 when SSO user has no Hybris account yet.
  Next.js treats 206 as "redirect to partial registration page". Auth tokens ARE returned in 206.
- **OAuth2 state parameter:** generated by Next.js on login initiation, validated on callback
  (CSRF protection). State stored in cookie between redirect hops.
- **Token storage:** access token in secure (but not httpOnly) cookie for client reads. Refresh
  token in httpOnly secure cookie.
- **Service M2M token:** `UserRegistrationService` obtains fresh client credentials token
  from IDP per registration call (no token caching).
- **Neo attributes** separated from standard user attributes in all models:
  `invitationCode`, `companyPosition`, `vatin`, `companyAddition`, `userType`.
- **Locale format normalization:** service converts `xx-YY` → `xx_YY` for IDP Accept-Language header.
- **Password masking in logs:** service replaces passwords with `*****` before logging request bodies.
- **Token refresh:** `POST /v1/oauth2/refresh { refreshToken }` → returns new token pair.
- **Logout:** `POST /v1/oauth2/logout` invalidates refresh token at IDP.

## Data Models

### `UserRegistrationRequest`
```
{
  userAttributes: {
    salutation, title?, firstName, lastName, email,
    password, language, country, company?,
    captchaResponse, termsAndConditions, privacyPolicy
  },
  neoAttributes?: {
    invitationCode?, companyAddition?, companyPosition?,
    userType?, vatin?
  }
}
```

### `PartialRegistrationRequest`
```
{
  userAttributes: {
    salutation?, firstName?, lastName?, language?,
    company?, captchaResponse,
    termsAndConditions?, privacyPolicy?
  },
  neoAttributes?: { ... }
}
```

### `TokenResponse`
```
{
  access_token: string,
  refresh_token: string,
  expires_in: int,
  refresh_expires_in: int
}
```

### `ValidateInvitationCodeResponse`
```
{ invitationValid: bool, message?: string }
```

## Known Issues & Gotchas

- **206 status = partial registration, not an error:** frontend must check for 206 specifically
  and redirect. Any other 2xx = success. 4xx/5xx = error.
- **reCAPTCHA bypass env var:** `X-Bypass-Captcha-Key` must match exactly. Different from
  other env var naming. Check `.env` / Secret Manager for exact key name.
- **Anonymous cart merge happens in `/api/auth/callback`:** if cart merge fails, it's logged
  but the login succeeds. Users may need to re-add items.
- **Token not validated with RSA in some services:** `ReadJwtToken()` without signature check
  is used for claim extraction in some places. This is intentional (service-internal use),
  but don't rely on it for security-critical decisions without proper validation.
- **Locale format conversion:** `de-DE` → `de_DE` for IDP. `en-GB` → `en_GB`. The underscore
  format is IDP/Hybris internal; the hyphen format is what XMCloud and the frontend use.

## Common Ticket Patterns

- **New Neo attribute in registration:** add to `NeoAttributes` model, update IDP mapper in
  `UserRegistrationService`, update registration form in frontend.
- **New required field:** add `[Required]` annotation to `UserAttributes`/`NeoAttributes` model.
- **reCAPTCHA threshold change:** update `RecaptchaApi` configuration (score threshold).
- **New OAuth2 callback action** (e.g., post-login redirect logic): update `callback/route.ts`.
- **B2B invitation code validation:** `POST /api/user/validate-invitation-code` → `POST /v1/validateInvitationCode/{code}`.
