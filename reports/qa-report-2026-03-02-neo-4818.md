## Implementation Review

### Correctness

- The EAN regex `\b\d{13}\b` correctly matches exactly 13 consecutive digits at word boundaries, distinguishing EANs from SKUs (which contain letters).
- `GetFinishesOrderedByEan()` correctly parses the string EAN to `long` via `TryParse` before comparing with `FinishDefinition.Ean` (which is `long`).
- The ordering pipeline places EAN ordering after SKU ordering: `Default -> SKU -> EAN -> Size -> PriceGroup -> Color -> Historical`. This means if both SKU and EAN match different variants, EAN takes final priority for the front position. This is documented and tested in `GetOrderedFinishes_SkuTakesPriority_WhenBothSkuAndEanProvided`.
- Both the search controller and autosuggestion controller wire EAN extraction and pass it through correctly.

### Design Quality

- Follows existing patterns exactly (same looping/reordering style as SKU, color, size ordering).
- EAN is an optional parameter with default `null` -- backward compatible.
- The `Ean` property on `IndexProductFieldsVariant` is properly included in equality and hash code.
