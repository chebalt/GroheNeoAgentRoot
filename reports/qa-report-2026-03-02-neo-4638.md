## 3. Bugs Found

### BUG-1: Dictionary service called with empty string when Lang is empty (SEVERITY: LOW)

**Location:** `SpecificationDocumentService.cs`, line 222

**Description:**
The fallback logic (`productLocale`) is applied to product API calls but NOT to the dictionary service call. When `Lang` is empty or null:

```csharp
var documentLang = request.SpecificationDocumentOptions.Lang;  // empty string
var productLocale = !string.IsNullOrEmpty(documentLang) ? documentLang : request.Locale;  // falls back correctly

// Products: uses productLocale (CORRECT)
await _productsApi.GetProductByIdAsync(sku, productLocale, false, 3);

// Dictionary: uses documentLang directly (BUG -- passes empty string)
await _dictionaryService.GetKeyValuePairs(documentLang);  // should be productLocale
```

**Impact:** When `Lang` is empty/null, the dictionary service receives an empty string instead of the fallback locale. Depending on how `IDictionaryItemsService.GetKeyValuePairs()` handles an empty string, this could result in:
- Missing dictionary labels in the PDF
- An XM Cloud GraphQL query with an invalid/empty language parameter

**Severity assessment:** LOW in practice because the frontend always sets `specificationFormValues.language` from the dropdown selection, so `Lang` should never be empty in normal operation. However, it is a defensive programming gap.

**Reproduction:**
Send a `CreateSpecificationDocumentRequest` where `SpecificationDocumentOptions.Lang = ""` and `Locale = "de-AT"`. The dictionary call will use `""` instead of `"de-AT"`.

**Recommendation for dev agent:**
Change line 222 from:
```csharp
projectListSpecificationDocument.Dictionary = await _dictionaryService.GetKeyValuePairs(documentLang);
```
to:
```csharp
projectListSpecificationDocument.Dictionary = await _dictionaryService.GetKeyValuePairs(productLocale);
```

Also update the log message on line 215 to use `productLocale` for consistency.

The existing test `CreateSpecificationDocument_FallsBackToRequestLocale_WhenLangIsEmpty` should be extended to also verify:
```csharp
_dictionaryItemsServiceMock.Verify(x => x.GetKeyValuePairs(requestLocale), Times.Once);
```

## 4. Known Limitation: Sustainability Section (Page 4)

**Confirmed:** The sustainability section content (title, introduction, left/right columns, image) is fetched from Sitecore XM Cloud at page render time using the page locale -- NOT the user-selected export language. This content is passed from the server component (`ExportProjectList.server.tsx`, lines 167-172) through the form to the mutation and then to the backend as-is.

The backend does NOT re-fetch sustainability content in the export language. It simply passes through whatever the frontend sent in the `sustainabilitySection` field.

**If "page 4" in the ticket refers to the sustainability section:** This remains unfixed. The sustainability text will always appear in the page/market language regardless of the export language selection.

**Root cause:** The sustainability content is sourced from CMS page props (SSR), not from dictionary items that can be fetched per-language by the backend.

**Possible fix approaches (out of scope for this PR):**
1. Move sustainability text to XM Cloud dictionary items so the backend can fetch them per-language (same as other PDF labels)
2. Have the frontend re-fetch the sustainability content in the selected language before submitting the export request
3. Have the backend fetch the sustainability content from XM Cloud GraphQL in the target language

## 5. Implementation Correctness Assessment

### Data flow (happy path -- user on de-AT selects en-GB as export language):

1. Frontend mutation: `documentLocale = specificationFormValues.language` = `"en-GB"`
2. BFF route receives: `locale=de-AT`, `documentLocale=en-GB`
3. BFF constructs body: `locale: "en-GB"` (from `documentLocale || locale`)
4. BFF fetches project list with page locale `de-AT` (correct -- list is market-scoped)
5. BFF fetches prices with page locale `de-AT` (correct -- prices are market-scoped)
6. Backend service receives: `request.Locale = "en-GB"`, `request.SpecificationDocumentOptions.Lang = "en-GB"`
7. Backend computes: `productLocale = "en-GB"` (from Lang)
8. Product data fetched in `en-GB` (CORRECT)
9. Dictionary fetched in `"en-GB"` (CORRECT for this path)

### Edge case -- Lang empty:
Steps 1-6 same but with `Lang = ""`
7. `productLocale = request.Locale` (fallback -- correct for products)
8. Dictionary called with `""` (BUG -- should use `productLocale`)

### Pricing locale:
The BFF route fetches prices using the page locale (`locale`), not `documentLocale`. This is correct because prices are market-specific (de-AT prices vs en-GB prices are different based on market, not language).

### Project list fetch locale:
The BFF fetches the project list using the page locale. This is correct because project lists are stored per-market.
